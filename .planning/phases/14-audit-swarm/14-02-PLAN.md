**Frontmatter:**
```yaml
name: skippy:review
description: Multi-agent audit swarm -- spawns specialist reviewers, aggregates findings, applies fixes, verifies
```

**Objective:** Run a multi-agent code review cycle on the current project or a specified scope. Spawns 4 specialist reviewers sequentially, aggregates findings into a shared board, applies fixes for critical issues, and verifies no regressions.

**Execution context:** Reference these files:
```
@../SKILL.md
@../references/audit-swarm.md
@../references/verification-loops.md
@../references/model-routing.md
```

**Process section** -- 8 steps matching the orchestration flow from audit-swarm.md:

**Step 1: Determine Scope and Prepare**
- If user specifies a scope (e.g., "review phase 13", "review tools/"), use that
- Otherwise, detect: check `.planning/STATE.md` for current phase, default to full repo scan
- Accept `--scope <path>` for targeted review of specific directories or files
- Create `.reports/skippy-review/` directory if it doesn't exist
- Create findings board file at `.reports/skippy-review/findings-{YYYY-MM-DD-HHMMSS}.md` with the template header (scope, timestamp, status: in-progress, empty sections for each reviewer)

**Step 2: Spawn Reviewer Agents (Sequential)**
- For each of the 4 reviewers (security, code-quality, architecture, consistency):
  1. Spawn the agent using the Agent tool, passing a prompt that includes:
     - The scope to review (directory path or file list)
     - The findings board file path (agent writes findings there)
     - Instruction: "Write your findings to the `## {Reviewer Name} Review` section of the findings board at {path}. Return a summary count only (e.g., 'Found 2 CRITICAL, 3 HIGH, 1 MEDIUM issues')."
  2. Wait for completion before spawning the next reviewer
  3. Record the summary count returned by each reviewer
- IMPORTANT: Spawn sequentially, NOT in parallel. This prevents context overflow from large simultaneous returns and avoids findings board write conflicts.

**Step 3: Synthesize Findings**
- Read the full findings board file
- Cross-reference findings between reviewers (e.g., same file flagged by both security and code-quality)
- Deduplicate findings that describe the same issue from different perspectives
- Add cross-ref notes to duplicate findings

**Step 4: Prioritize**
- Build the Priority Actions table in the Synthesis section
- Sort by severity: CRITICAL first, then HIGH, then MEDIUM, then LOW
- Group by file for efficient fix planning (multiple findings on one file = one fix agent)
- Count totals by severity and report to user

**Step 5: Fix (CRITICAL and HIGH only)**
- For each CRITICAL or HIGH finding (or group of findings on the same file):
  1. Spawn the fix-agent with a prompt containing:
     - The specific finding(s) to fix (copy from findings board)
     - The file path(s) to modify
     - Instruction: "Apply the fix. Make one atomic commit per finding with message prefixed `fix(review):`. Report the commit hash and what was changed."
  2. Wait for completion
  3. Record commit hash and fix description in the Fix Log section of findings board
- If no CRITICAL or HIGH findings, skip this step and report "No critical fixes needed"

**Step 6: Evaluate**
- Spawn the eval-agent with a prompt containing:
  - The findings board path (to see what was fixed)
  - The list of commit hashes from Step 5
  - Instruction: "Verify each fix was applied correctly. Check that fixes didn't introduce regressions. Write results to the `## Evaluation` section. Return PASS or FAIL with details."
- If eval returns FAIL:
  - Check if the failure matches a previous failure signature (same-failure detection from verification-loops.md)
  - If same failure 3 times: stop cycling, report the persistent issue
  - If new failure and cycle count < 3: spawn a targeted fix-agent for the regression, then re-evaluate
  - If cycle count >= 3: stop cycling, report remaining issues

**Step 7: Finalize Findings Board**
- Update the findings board status to "verified" (or "partial -- N issues remain")
- Add final statistics: total findings by severity, fixes applied, regressions found/resolved
- Ensure all finding statuses are updated (OPEN -> FIXED or WONTFIX with reason)

**Step 8: Report**
- Display a summary to the user:
  ```
  ## Audit Complete

  **Scope:** {what was reviewed}
  **Findings:** {N} total ({C} critical, {H} high, {M} medium, {L} low)
  **Fixed:** {F} findings across {X} commits
  **Regressions:** {R} found, {RR} resolved
  **Status:** {CLEAN | PARTIAL | NEEDS_ATTENTION}

  Full report: .reports/skippy-review/findings-{timestamp}.md
  ```
- If NEEDS_ATTENTION: list the remaining unfixed CRITICAL/HIGH items

**Important notes to include in the command:**
- This command does NOT require GSD or any external framework
- The main conversation acts as orchestrator -- all agent spawns happen here
- Subagents CANNOT spawn other subagents (Claude Code limitation)
- All agents include HOME sandbox instructions -- never operate against real ~/.claude/
- Sequential spawning is intentional (see audit-swarm.md for rationale)
