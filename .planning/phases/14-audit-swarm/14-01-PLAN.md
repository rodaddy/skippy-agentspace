Create `skills/skippy/agents/` directory and 6 agent markdown files. Each file uses the Claude Code subagent format: YAML frontmatter with `name`, `description`, `tools`, `model`, `permissionMode` fields, followed by a system prompt body.

**All 4 reviewer agents** share this structure pattern:
- `tools: Read, Grep, Glob, Bash` (read-only operations; Bash needed for grep/find/test commands)
- `model: sonnet` (MEDIUM complexity per model-routing.md -- review is standard analysis work)
- `permissionMode: plan` (enforces read-only; prevents accidental writes)
- System prompt includes: role description, focus areas (unique per agent), sandbox rule (`export HOME=$(mktemp -d)` before any HOME-referencing operation), output format (findings in structured markdown with severity/file/type/evidence/fix/cross-ref fields)

**security-reviewer.md** focuses on: secret/credential exposure, shell injection vectors (eval, unquoted vars), path traversal, unsafe file operations (unguarded rm, chmod 777), .gitignore security gaps.

**code-quality-reviewer.md** focuses on: code duplication/DRY violations, error handling gaps (missing set -e, unchecked return codes), dead code, overly complex functions (>50 lines), inconsistent naming, missing validation of inputs.

**architecture-reviewer.md** focuses on: skill portability violations (cross-skill imports, absolute paths), convention violations (CONVENTIONS.md rules), circular dependencies, separation of concerns (scripts doing too much), consistency with the slim SKILL.md + deep references pattern. Use `model: opus` for this agent -- architecture analysis requires HIGH complexity reasoning per model-routing.md.

**consistency-reviewer.md** focuses on: SKILL.md accuracy (do listed files exist?), INDEX.md staleness, marketplace.json accuracy, command descriptions matching actual behavior, cross-file state alignment (STATE.md vs ROADMAP.md vs actual files).

**fix-agent.md:**
- `tools: Read, Grep, Glob, Bash, Write, Edit` (needs write access to apply fixes)
- `model: sonnet` (standard fixes; orchestrator can override to opus for complex ones)
- `permissionMode: bypassPermissions` (autonomous fixing)
- `isolation: worktree` (git worktree isolation prevents conflicts)
- System prompt includes: role (apply specific fixes from findings), atomic commit rule (one commit per finding, descriptive message prefixed `fix(review):`), sandbox rule, instruction to report back what was fixed and commit hash

**eval-agent.md:**
- `tools: Read, Grep, Glob, Bash` (read-only for verification)
- `model: sonnet`
- `permissionMode: plan` (read-only)
- System prompt includes: role (verify fixes applied correctly, check for regressions), cycling protocol (reference verification-loops.md -- max 3 iterations, same-failure detection), output format (PASS/FAIL per fix, regression list, overall verdict)

Create `skills/skippy/references/audit-swarm.md` following the established reference doc pattern (see existing references like verification-loops.md for style: heading, intro paragraph, source table, "Why This Version", pattern sections, integration points, "When to Apply", footer with sources).

The reference doc defines the audit swarm orchestration protocol with these sections:

**Header:** "Audit Swarm -- Multi-Agent Review Protocol" with intro paragraph explaining the pattern (sequential specialist review, shared findings board, fix/eval cycling).

**Source Upstreams table:** Source is the v1.1 audit process (7 rounds, 20+ agents, 17 findings). Strength: proven at scale. Weakness: was ad-hoc, not codified.

**Why This Version:** Codifies the ad-hoc v1.1 audit into a repeatable command. Key insight: the orchestrator (main conversation) manages all agent spawning because subagents cannot spawn subagents.

**Pattern sections:**

1. **Orchestration Flow** -- The 8-step process: scope detection, preparation (create findings board), sequential reviewer spawning (4 agents), synthesis (cross-reference, deduplicate), prioritization (severity sort, file grouping), fix agent spawning (per-finding or per-file), evaluation (regression check, max 3 cycles), reporting (final audit report with stats).

2. **Findings Board Protocol** -- The shared markdown file structure at `.reports/skippy-review/findings-{timestamp}.md`. Sections: header (scope, timestamp, status), per-reviewer sections (Security Review, Code Quality Review, Architecture Review, Consistency Review), Synthesis section with priority actions table (columns: #, Severity, Finding, Reviewer, Fix Status), Fix Log (commit hashes), Evaluation section.

3. **Severity Classification** -- CRITICAL (must fix, security/data loss), HIGH (should fix, logic errors), MEDIUM (recommended, quality), LOW (optional, style). Only CRITICAL and HIGH trigger fix cycles.

4. **Sandbox Protocol** -- Three layers: HOME override in system prompt, worktree isolation for fix agents, tool restrictions (reviewers get read-only). Reference the 71-skill nuke incident as motivation.

5. **Agent Roster** -- Table listing all 6 agents with their role, model tier, permission mode, and tool set.

6. **Exit Conditions** -- When to stop cycling: all CRITICAL/HIGH fixed, max 3 eval cycles reached, same failure 3 times (escalate per verification-loops.md), environment error.

**Integration Points:** Links to verification-loops.md (cycling protocol), model-routing.md (tier selection), checkpoints.md (human verification gates), plan-structure.md (task format for reporting).

**When to Apply:** After phase completion, before milestone release, ad-hoc on specific files/directories, after major refactoring.

**Footer:** Sources line crediting v1.1 audit process. "Last reviewed: {today's date}"

Keep to ~120-150 lines. This is a reference doc, not a novel.
