# Debugging Subagents

When a subagent goes sideways, the JSONL transcript is your primary diagnostic tool. `/trace` shows session-level event flow -- this guide covers drilling into a specific subagent's conversation to find where things broke.

## When to Use This

- A subagent returned a bad summary or incomplete results
- An agent silently failed (no error, just wrong output)
- You need to understand why a delegation decision was made
- `/trace` shows the agent ran but the outcome was wrong

## Where Transcripts Live

```
~/.claude/projects/{project}/{sessionId}/subagents/agent-{agentId}.jsonl
```

Each line is a JSON object. Find the right file by timestamp -- most recently modified is usually the target.

## Message Types

| Type | What It Tells You |
|------|------------------|
| `assistant/thinking` | The subagent's reasoning chain -- what it planned to do |
| `assistant/tool_use` | Actions taken -- which tools, what parameters |
| `user/tool_result` | What actually happened -- success, errors, unexpected output |
| `assistant/text` | Final summary returned to the main agent |

## Diagnostic Process

### 1. Find the Gap

The bug lives in the **gap between what the subagent planned and what occurred**. Read `thinking` entries to see intent, then compare against `tool_result` entries for reality.

```bash
# Quick scan -- show all thinking and tool results
cat ~/.claude/projects/*/$(ls -t ~/.claude/projects/*/subagents/ 2>/dev/null | head -1) 2>/dev/null | \
  jq -r 'select(.type == "assistant/thinking" or .type == "user/tool_result") | .type + ": " + (.content // .text // "" | tostring)[:120]'
```

### 2. Check Delegation Accuracy

Was the right subagent chosen? If an Explore agent was delegated a task requiring file modifications, the problem is upstream -- the task description didn't match the agent's capabilities.

**Signs of wrong delegation:**
- `tool_use` entries showing attempts to call tools the agent doesn't have
- `tool_result` errors like "tool not available" or "permission denied"
- The agent working around a missing tool instead of reporting the limitation

### 3. Review Tool Usage

Check `tool_use` entries for:
- Unexpected parameters (wrong file paths, malformed inputs)
- Repeated calls to the same tool (agent stuck in a loop)
- Tools called in the wrong order (writing before reading)

### 4. Examine Tool Results

`tool_result` entries with errors point directly to root cause. Common culprits:

| Error Pattern | Likely Cause |
|---------------|-------------|
| File not found | Wrong path assumption -- agent didn't explore first |
| Permission denied | Agent type too restrictive for the task |
| Timeout | Command took too long -- break into smaller chunks |
| Empty result | Search pattern too narrow or file doesn't exist |
| Hook blocked | LAW violation -- agent tried something prohibited |

### 5. Evaluate the Summary

The subtlest failure mode: the agent did good work but returned a poor summary. The main agent then made bad decisions based on incomplete information.

**Check:** Does the final `assistant/text` entry capture all the important findings from the tool results? If the agent read 15 files but only mentioned 3 in the summary, the main agent is working with 20% of the data.

## Common Patterns

### Resume, Don't Restart

If a subagent got partway through before hitting an issue, resume it with `SendMessage` using the stored `agentId`. Restarting from scratch wastes accumulated context and costs tokens.

```
SendMessage(to: agentId, content: "The file moved to src/new/path.ts -- continue from where you left off")
```

### Circuit Breaker

If an agent fails the same way 3 times, stop retrying. The problem is in the delegation (wrong agent type, missing context, or task too ambiguous). Re-examine the prompt.

### Summary Quality Fix

When an agent does good work but returns a vague summary, add explicit output format requirements to the agent's prompt:

```
Return a structured summary with:
1. Files examined (paths)
2. Key findings (one line each)
3. Recommended actions (specific, actionable)
```

## Relationship to /trace

| Tool | Scope | Data Source |
|------|-------|-------------|
| `/trace` | Session-level timeline | `.skippy/trace.jsonl` (explicitly written by skills) |
| This guide | Single subagent conversation | `~/.claude/projects/.../agent-{id}.jsonl` (auto-generated by Claude Code) |

Use `/trace` to find which agent went wrong. Use this guide to find why.
